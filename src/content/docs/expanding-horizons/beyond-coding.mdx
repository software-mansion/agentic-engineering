The previous sections focused on agents writing, reviewing, and shipping code.
But a software project involves a lot of work that is not code: triaging issues, answering support questions, synthesizing meeting notes or responding to production incidents.
Agents handle all of these, but the same principles from the previous chapters apply, mainly — give them good context and verify their replies.

## Issue triage

Incoming issues accumulate fast.
Deciding which ones are duplicates, which are actionable, and which need more information is a great task for agents, as the work is tedious and pattern-heavy.

A triage agent connected to your issue tracker can:
- Tag and label issues automatically based on title and description
- Identify duplicates and link them to existing reports
- Ask clarifying questions in a comment before routing to a team member
- Prioritize by severity, user impact, or affected component

For example, OpenAI uses scheduled triage automations for exactly this, as covered in the [High-level harnesses](/expanding-horizons/high-level-harnesses/) chapter.
The agent runs on a fixed schedule, processes new issues, and only escalates what really needs a human decision.

:::tip
Start with read-only triage: the agent labels and comments, but never closes or assigns.
Once you have validated its judgment on a few hundred issues, gradually give it more authority.
:::

## Customer support

Support queues are high-volume and often repetitive.
Agents can handle a large share of incoming tickets by relying on documentation, changelogs, and past resolved tickets as context.

Things agents handle well:
- Answering "how do I..." questions against product docs
- Collecting structured reproduction steps for bug reports before escalating
- Drafting responses to common error messages or known limitations
- Translating support tickets into structured bug reports for engineering

Things to watch:
- Agents should escalate when uncertain, not guess
- Every outgoing message should be attributable — customers should know when they are talking to an automated system

:::caution
Do not let a support agent close tickets autonomously until you have built confidence in its judgment.
A closed ticket a customer considers unresolved is worse than a slow response.
:::

## Meeting notes and action items

Meetings produce commitments that scatter across memory, Slack threads, and inbox tabs.
Agents connected to transcription services can close this loop automatically.

A typical setup:
1. A transcription tool (Otter, Fireflies, Grain, or similar) produces a raw transcript.
2. An agent reads the transcript and extracts: decisions made, open questions, and action items with owners.
3. The output lands in a shared doc, a project management tool, or as comments on relevant issues.

The value is not a summary — it is the structured extraction.
A good action item note names a person, a task, and a deadline.
An agent is better at finding these in a noisy transcript than a person skimming it two days later.

:::note[Remember]
Agents work best here when the meeting has a clear structure — standups, retros, design reviews.
For exploratory or creative discussions, the raw transcript often carries information that a summary loses.
:::

## Observability and incident response

Production incidents are time-sensitive and require pulling context from many places at once: error logs, stack traces, recent deployments, related tickets.
Agents can do a significant share of this triage automatically, before any engineer has responded.

Agents wired into the observability loop can:
- Read a stack trace, search the codebase for the relevant code path, identify the root cause, and propose a fix
- Summarize the blast radius of a deployment ("these three endpoints regressed after this PR")
- Correlate errors with recent commits and flag the likely culprit
- Draft incident post-mortems from collected timeline data
- Annotate alerts with human-readable explanations, not just threshold values

The engineer still reviews and approves changes, but the expensive initial investigation step happens automatically.

:::tip
Wire agents into your alerting pipeline at the notification stage.
An alert that arrives with a preliminary root-cause summary is significantly easier to act on than a raw error rate chart.
:::

## Release notes and changelogs

Writing release notes is easy to skip and hard to do well under deadline.
Agents can generate a first draft from merged PRs and commit messages, filtered and grouped by audience.

The output quality depends heavily on what you give the agent to work with:
- PR titles and descriptions that are consistently written — or pre-processed by a PR review agent
- A template that defines what a release note should contain
- An audience definition: internal release notes and customer-facing changelogs have different tones and levels of detail

The result usually needs editing, but editing is much faster than writing from scratch.

## Dependency and security hygiene

Keeping dependencies up to date and catching known vulnerabilities requires constant vigilance that most teams deprioritize.
Scheduled bots open PRs for dependency updates on a fixed schedule.
An agent can review whether a major-version bump is safe given your actual usage of the library before you spend time on it.
Security scanners can be wired to an agent that triages CVEs and drafts upgrade paths.

The safe pattern is agent-drafted PRs verified by CI and reviewed by a human — not unattended upgrades straight to main.
